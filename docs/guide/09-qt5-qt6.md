## 一、核心变化

### Q1: ⭐🟢 Qt6 相比 Qt5 的总体变化是什么？


A: 结论：Qt6 整体更现代化，构建系统更偏向 CMake，图形后端更抽象统一，很多 API 也更贴近现代 C++。如果一句话总结，就是：Qt6 是 Qt 面向未来工具链和平台的一次系统升级。


详细解释：


- 强化 CMake 作为主流构建方案。
- 图形层引入 RHI 抽象。
- 字符串/容器/view 类更现代。
- 部分旧模块和过时 API 被清理。


常见坑/追问：


- Qt6 不是“Qt5 小修小补”，迁移时要认真做兼容评估。

### Q2: ⭐🟡 Qt6 为什么更强调 CMake？


A: 结论：因为 CMake 已成为 C++ 生态事实标准，跨平台集成第三方库、IDE、CI/CD 都更统一。Qt6 强化 CMake，本质是在拥抱主流工程生态。


详细解释：


- qmake 仍有历史价值，但生态势能已弱。
- CMake 更利于大型多模块工程和非 Qt 项目混编。
- 官方文档和模板也逐步向 CMake 倾斜。


常见坑/追问：


- 面试中别说成“qmake 完全不能用了”，重点是未来趋势和主流支持方向。

> 💡 **面试追问**：模板编译期展开有什么代价？如何减少模板实例化导致的代码膨胀？



### Q3: ⭐🟡 Qt6 的图形渲染架构变化是什么？


A: 结论：Qt6 更强调 RHI（Rendering Hardware Interface）抽象层，把渲染后端从“偏 OpenGL”升级为能适配 Vulkan、Metal、D3D、OpenGL 的统一接口。这让跨平台图形能力更灵活。


详细解释：


- 不同平台最佳图形 API 不一样。
- RHI 降低了上层对单一图形 API 的耦合。
- 对 Qt Quick 场景尤其重要。


常见坑/追问：


- 不是所有普通 Widgets 程序都会明显感知到这层变化，但底层方向很关键。

### Q4: 🟡 QRegExp 为什么被 QRegularExpression 替代？


A: 结论：因为 `QRegularExpression` 在性能、语义和与现代正则标准兼容性上都更好，旧的 `QRegExp` 已经带着明显历史包袱。新代码应优先使用新类。


详细解释：


- 新类 API 更现代。
- 与 Perl 兼容正则能力更接近主流期待。
- 迁移时要注意匹配行为差异。


常见坑/追问：


- 老项目迁移正则时要补测试，别想当然认为结果完全一致。

### Q5: ⭐🟡 Qt6 里 QList 和 QVector 有什么关键变化？


A: 结论：Qt6 中 `QList` 的底层实现改成更接近连续存储，与 `QVector` 的差异大幅缩小，因此很多旧时代关于 `QList` 性能特性的经验需要更新。别再背老黄历了。


详细解释：


- Qt5 时代两者差异更明显。
- Qt6 里容器选择更多回到接口语义和历史兼容，而不是极端性能差异。


常见坑/追问：


- 面试若只会背“QList 链表、QVector 数组”，在 Qt6 语境下容易翻车。

## 二、模块与 API 迁移

### Q6: 🟡 Qt6 在字符串和 view 类接口上有什么现代化改进？


A: 结论：Qt6 更强调 view 类，如 `QStringView`、`QLatin1StringView`，目标是减少不必要拷贝，让接口表达“只读视图”而不是强制所有权。思路和标准库 `string_view` 很接近。


详细解释：


- 更高效的只读参数传递。
- 更清晰地区分“拥有数据”和“观察数据”。
- 也意味着生命周期问题需要更注意。


常见坑/追问：


- view 类和 `string_view` 一样，都有悬空视图风险。

### Q7: 🟡 Qt6 的高 DPI 支持为什么更省心？


A: 结论：因为 Qt6 在高 DPI 支持上默认行为更现代，适配多缩放、多分辨率显示器的体验通常更好。过去 Qt5 中一些需要手工配置的步骤，在 Qt6 中更自然了。


详细解释：


- 桌面环境已普遍进入混合 DPI 时代。
- 新默认策略减少模糊、错位和缩放比例异常。


常见坑/追问：


- 不同平台窗口系统表现仍可能不同，不能说“完全无坑”。

### Q8: ⭐🟡 Qt5 项目迁移 Qt6 时最容易踩哪些坑？


A: 结论：高频坑包括构建系统调整、废弃 API 移除、模块拆分变化、第三方依赖兼容性以及容器/字符串行为差异。迁移真正难的不是编译通过，而是行为回归验证。


详细解释：


- 先看官方 porting guide。
- 再清理过时 API。
- 最后重点做 UI、渲染、输入法、高 DPI、插件加载等回归测试。


常见坑/追问：


- “能跑起来”不等于迁移完成，很多坑在客户环境机器上才暴露。

### Q9: ⭐🟡 新项目什么时候更推荐 Qt6？


A: 结论：只要目标平台、编译器和依赖链允许，新项目通常更推荐 Qt6，因为维护周期更长、生态方向更明确、底层架构也更现代。除非有明显历史包袱，否则没必要主动站在旧船上。


详细解释：


- Qt6 是官方未来主线。
- 新平台支持和长期维护更优。
- 团队若还要持续迭代多年，越早上主线越划算。


常见坑/追问：


- 如果你的项目强依赖某些仅 Qt5 成熟的第三方组件，就要实事求是评估。

### Q10: 🟡 什么情况下还会继续用 Qt5？


A: 结论：老项目稳定优先、平台/编译器受限、认证流程严格或第三方库迁移成本过高时，继续用 Qt5 是合理选择。工程决策不是只看新旧，还要看风险和收益。


详细解释：


- 医疗、工控、国企环境里“稳定压倒一切”很常见。
- 老项目如果没有明显升级收益，贸然迁移可能得不偿失。


常见坑/追问：


- 回答时不要表现得“守旧就是落后”，重点是生命周期和成本控制。

## 三、性能与兼容性

### Q11: ⭐🟡 Qt6 中 QML 有哪些重要变化？


A: 结论：Qt6 的 QML 引入了强类型（Type System）、模块化（QML Module）、CMake 集成和编译期优化；`qt_add_qml_module` 替代了旧的 `qmldir` 手动管理，静态分析和 IDE 支持大幅提升。


详细解释：


- **强类型 QML**：属性可声明类型（`property int count`），减少运行时类型错误。
- **QML 模块**：`qt_add_qml_module()` CMake 宏自动生成 `qmldir`，管理命名空间和版本。
- **编译期 QML**：`QML_ELEMENT` 宏 + 编译期类型注册，替代 `qmlRegisterType`，性能更好。
- **QML Linter（qmllint）**：Qt6 大幅增强，可检测类型错误、未定义属性等，集成 CI。


代码示例：


```cmake
qt_add_qml_module(myapp
    URI "com.myapp"
    VERSION 1.0
    QML_FILES Main.qml
    SOURCES mybackend.cpp mybackend.h
)
```


常见坑/追问：


- Qt5 项目迁移时 `qmlRegisterType` 的 URI 和 CMake 模块 URI 必须一致，否则 import 找不到。
- 追问：`QML_ELEMENT` 和 `qmlRegisterType` 的区别？前者编译期注册，错误在编译时暴露；后者运行时注册，错误在运行时才发现。

### Q12: ⭐🟡 Qt6 对 OpenGL/Vulkan/Metal 渲染后端有什么变化？


A: 结论：Qt6 引入 RHI（Rendering Hardware Interface）抽象层，Qt Quick 渲染默认通过 RHI，支持 OpenGL、Vulkan、Metal、Direct3D 11/12，不再直接绑定 OpenGL，跨平台渲染更统一。


详细解释：


- Qt5：Qt Quick 强依赖 OpenGL，macOS 上 OpenGL 被 Apple 废弃后问题明显。
- Qt6 RHI：渲染调用统一经过 `QRhi` 抽象层，后端可按平台自动选择（macOS→Metal，Windows→D3D11，Linux→Vulkan/OpenGL）。
- 自定义渲染：用 `QRhiCommandBuffer` 写渲染代码，或继续用 `QOpenGLWidget`（仍支持）。
- 性能：Metal/Vulkan 后端在对应平台比 OpenGL 有明显提升。


常见坑/追问：


- 强制指定后端：`QSG_RHI_BACKEND=vulkan ./app` 环境变量。
- 追问：Qt5 的 `QOpenGLFunctions` 在 Qt6 还能用吗？能，但新项目推荐用 RHI，避免平台差异。

### Q13: ⭐🟡 Qt6 的 CMake 迁移有哪些主要变化？


A: 结论：Qt6 官方推荐 CMake，废弃 qmake 维护（仍可用）；`find_package(Qt6)` + `qt_add_executable` + `target_link_libraries` 替代 qmake 的 `.pro` 文件，模块化更清晰。


详细解释：


- `qt_add_executable`：封装了 `add_executable` + 平台特定处理（如 Windows 的 `WIN32` 参数）。
- `target_link_libraries(app PRIVATE Qt6::Core Qt6::Widgets)`：按需链接，不需要全量导入。
- Qt6 CMake 集成了 MOC、RCC、UIC 自动处理，不需要手动配置 `AUTOMOC` 等。
- `qt_add_qml_module` / `qt_add_resources` 封装了资源和 QML 管理。


代码示例：


```cmake
find_package(Qt6 REQUIRED COMPONENTS Core Widgets)
qt_add_executable(myapp main.cpp)
target_link_libraries(myapp PRIVATE Qt6::Core Qt6::Widgets)
```


常见坑/追问：


- qmake 的 `INCLUDEPATH`/`LIBS` 对应 CMake 的 `target_include_directories`/`target_link_libraries`。
- 追问：Qt6 CMake 和手写 CMake 有什么区别？Qt6 提供的宏封装了平台差异，推荐使用；纯 CMake 写法也可以但需手动处理更多细节。

### Q14: ⭐🔴 从 Qt5 迁移到 Qt6 最容易踩的坑是什么？


A: 结论：最常见的坑是：废弃 API（`QRegExp` → `QRegularExpression`，`QString::sprintf` 等）、信号槽语法变化（新式语法在 Qt6 更严格）、QML 模块化要求和字符集处理变化。


详细解释：


- `QRegExp` 被移除（Qt5 已废弃），必须改用 `QRegularExpression`。
- `Qt::AA_EnableHighDpiScaling` 等属性 Qt6 默认开启，设置反而报错。
- `QVariant::canConvert<T>()` 行为变化，某些隐式转换被移除。
- `QString` 操作符变化：Qt6 禁止 `char*` 直接赋值给 `QString`，必须用 `QString::fromUtf8` 等显式转换。
- CMake 最低版本要求提高（Qt6 需要 CMake 3.16+）。


常见坑/追问：


- 升级前运行 `qt5compat` 模块可以暂时兼容，但不是长期解决方案。
- 追问：有没有自动迁移工具？`clazy` 静态分析器可以检测 Qt5 废弃用法；官方迁移指南提供了完整清单。

### Q15: ⭐🟡 Qt6 的 Permission API 和平台权限管理是什么？


A: 结论：Qt6.5 引入 `QPermission` API，统一了 Android/iOS/macOS 等平台的权限请求接口，C++ 代码无需再针对不同平台手写 JNI 或 Objective-C 权限请求。


详细解释：


- `qApp->requestPermission(QCameraPermission{}, ...)` 统一语法，回调中判断是否授权。
- 支持的权限类型：Camera、Microphone、Bluetooth、Contacts、Calendar、Location 等。
- 在 Android 上自动处理 `onRequestPermissionsResult` 回调，在 macOS 上调用系统 API。
- 桌面 Linux 权限模型不同，相关 API 在 Linux 上通常直接返回 Granted。


代码示例：


```cpp
qApp->requestPermission(QCameraPermission{}, this, [](const QPermission& p) {
    if (p.status() == Qt::PermissionStatus::Granted) {
        startCamera();
    } else {
        qWarning("Camera permission denied");
    }
});
```


常见坑/追问：


- iOS/Android 还需要在 Info.plist / AndroidManifest.xml 中声明权限，Qt API 只处理运行时请求。
- 追问：Qt6.5 之前怎么处理 Android 权限？用 `QtAndroid::requestPermissions()` JNI 封装，代码较繁琐。
