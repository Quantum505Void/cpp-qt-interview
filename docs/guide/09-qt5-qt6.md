# 9. Qt5 与 Qt6 的区别


↑ 回到目录


### Q1: ⭐🟢 Qt6 相比 Qt5 的总体变化是什么？


A: 结论：Qt6 整体更现代化，构建系统更偏向 CMake，图形后端更抽象统一，很多 API 也更贴近现代 C++。如果一句话总结，就是：Qt6 是 Qt 面向未来工具链和平台的一次系统升级。


详细解释：


- 强化 CMake 作为主流构建方案。
- 图形层引入 RHI 抽象。
- 字符串/容器/view 类更现代。
- 部分旧模块和过时 API 被清理。


常见坑/追问：


- Qt6 不是“Qt5 小修小补”，迁移时要认真做兼容评估。


### Q2: ⭐🟡 Qt6 为什么更强调 CMake？


A: 结论：因为 CMake 已成为 C++ 生态事实标准，跨平台集成第三方库、IDE、CI/CD 都更统一。Qt6 强化 CMake，本质是在拥抱主流工程生态。


详细解释：


- qmake 仍有历史价值，但生态势能已弱。
- CMake 更利于大型多模块工程和非 Qt 项目混编。
- 官方文档和模板也逐步向 CMake 倾斜。


常见坑/追问：


- 面试中别说成“qmake 完全不能用了”，重点是未来趋势和主流支持方向。


### Q3: ⭐🟡 Qt6 的图形渲染架构变化是什么？


A: 结论：Qt6 更强调 RHI（Rendering Hardware Interface）抽象层，把渲染后端从“偏 OpenGL”升级为能适配 Vulkan、Metal、D3D、OpenGL 的统一接口。这让跨平台图形能力更灵活。


详细解释：


- 不同平台最佳图形 API 不一样。
- RHI 降低了上层对单一图形 API 的耦合。
- 对 Qt Quick 场景尤其重要。


常见坑/追问：


- 不是所有普通 Widgets 程序都会明显感知到这层变化，但底层方向很关键。


### Q4: 🟡 QRegExp 为什么被 QRegularExpression 替代？


A: 结论：因为 `QRegularExpression` 在性能、语义和与现代正则标准兼容性上都更好，旧的 `QRegExp` 已经带着明显历史包袱。新代码应优先使用新类。


详细解释：


- 新类 API 更现代。
- 与 Perl 兼容正则能力更接近主流期待。
- 迁移时要注意匹配行为差异。


常见坑/追问：


- 老项目迁移正则时要补测试，别想当然认为结果完全一致。


### Q5: ⭐🟡 Qt6 里 QList 和 QVector 有什么关键变化？


A: 结论：Qt6 中 `QList` 的底层实现改成更接近连续存储，与 `QVector` 的差异大幅缩小，因此很多旧时代关于 `QList` 性能特性的经验需要更新。别再背老黄历了。


详细解释：


- Qt5 时代两者差异更明显。
- Qt6 里容器选择更多回到接口语义和历史兼容，而不是极端性能差异。


常见坑/追问：


- 面试若只会背“QList 链表、QVector 数组”，在 Qt6 语境下容易翻车。


### Q6: 🟡 Qt6 在字符串和 view 类接口上有什么现代化改进？


A: 结论：Qt6 更强调 view 类，如 `QStringView`、`QLatin1StringView`，目标是减少不必要拷贝，让接口表达“只读视图”而不是强制所有权。思路和标准库 `string_view` 很接近。


详细解释：


- 更高效的只读参数传递。
- 更清晰地区分“拥有数据”和“观察数据”。
- 也意味着生命周期问题需要更注意。


常见坑/追问：


- view 类和 `string_view` 一样，都有悬空视图风险。


### Q7: 🟡 Qt6 的高 DPI 支持为什么更省心？


A: 结论：因为 Qt6 在高 DPI 支持上默认行为更现代，适配多缩放、多分辨率显示器的体验通常更好。过去 Qt5 中一些需要手工配置的步骤，在 Qt6 中更自然了。


详细解释：


- 桌面环境已普遍进入混合 DPI 时代。
- 新默认策略减少模糊、错位和缩放比例异常。


常见坑/追问：


- 不同平台窗口系统表现仍可能不同，不能说“完全无坑”。


### Q8: ⭐🟡 Qt5 项目迁移 Qt6 时最容易踩哪些坑？


A: 结论：高频坑包括构建系统调整、废弃 API 移除、模块拆分变化、第三方依赖兼容性以及容器/字符串行为差异。迁移真正难的不是编译通过，而是行为回归验证。


详细解释：


- 先看官方 porting guide。
- 再清理过时 API。
- 最后重点做 UI、渲染、输入法、高 DPI、插件加载等回归测试。


常见坑/追问：


- “能跑起来”不等于迁移完成，很多坑在客户环境机器上才暴露。


### Q9: ⭐🟡 新项目什么时候更推荐 Qt6？


A: 结论：只要目标平台、编译器和依赖链允许，新项目通常更推荐 Qt6，因为维护周期更长、生态方向更明确、底层架构也更现代。除非有明显历史包袱，否则没必要主动站在旧船上。


详细解释：


- Qt6 是官方未来主线。
- 新平台支持和长期维护更优。
- 团队若还要持续迭代多年，越早上主线越划算。


常见坑/追问：


- 如果你的项目强依赖某些仅 Qt5 成熟的第三方组件，就要实事求是评估。


### Q10: 🟡 什么情况下还会继续用 Qt5？


A: 结论：老项目稳定优先、平台/编译器受限、认证流程严格或第三方库迁移成本过高时，继续用 Qt5 是合理选择。工程决策不是只看新旧，还要看风险和收益。


详细解释：


- 医疗、工控、国企环境里“稳定压倒一切”很常见。
- 老项目如果没有明显升级收益，贸然迁移可能得不偿失。


常见坑/追问：


- 回答时不要表现得“守旧就是落后”，重点是生命周期和成本控制。
