# 34. 面试实战技巧


↑ 回到目录


### Q1: ⭐🟢 自我介绍怎么讲最稳？


A: 结论：控制在 1-2 分钟，围绕岗位匹配度讲“经验年限 + 核心技术栈 + 代表项目 + 你的优势”。


详细解释：


- C++/Qt 岗重点突出通信、并发、桌面应用、Linux、工程化。
- 结尾可自然引出后续追问点，比如 HID、OTA、性能优化。


常见坑/追问：


- 不要从大学经历一路讲到今天，像族谱展开。


### Q2: ⭐🟢 遇到不会的问题怎么办？


A: 结论：先说自己确定的部分，再说明不确定点和推理方向，别硬编。


详细解释：


- 面试官常看的是思维方式，不只是最终答案。
- 能把问题拆解出来，通常比瞎猜更加分。


常见坑/追问：


- “这个我没做过，但如果让我排查，我会先看……”是很好的转法。


### Q3: ⭐🟢 为什么面试里要多用量化表达？


A: 结论：量化能提高可信度，让成果从“感觉不错”变成“可以评估”。


详细解释：


- 如“启动时间从 6s 降到 2.5s”“日志丢失率降低到万分之一以下”。
- 量化越贴近业务，效果越好。


常见坑/追问：


- 别编数据；没有精确值就说“大约”“量级”。


### Q4: ⭐🟢 STAR 方法在技术面中怎么用？


A: 结论：STAR 不是只给 HR 用，技术面讲项目难点、线上问题、跨团队协作时同样非常好用。


详细解释：


- 尤其讲事故处理、性能优化、重构项目时结构非常清晰。
- Action 部分要体现你的技术判断，而非流水账。


常见坑/追问：


- 结果别只说“最后解决了”，要补“怎么证明解决了”。


### Q5: 🟡 技术面中如何体现“沟通靠谱”？


A: 结论：回答时先给结论，再展开；承认边界；把假设说清楚；别抢答和绕圈子。


详细解释：


- 面试官其实也在评估合作成本。
- 一个能把复杂问题讲清楚的人，通常更适合工程岗位。


常见坑/追问：


- 过度炫术语、答非所问、抢着证明自己都容易扣分。


### Q6: 🟡 反问面试官时可以问什么？


A: 结论：优先问技术栈、业务复杂度、团队协作方式、质量体系、版本节奏，而不是上来只问福利。


详细解释：


- 比如：主要设备通信协议有哪些？Qt5/6 比例？CI/CD 和代码评审怎么做？
- 这些问题能体现你真正在乎工作内容。


常见坑/追问：


- 别问官网上明明写着的问题。


### Q7: 🔴 如果面试官质疑你项目深度不够，怎么应对？


A: 结论：不要硬顶，立刻切到一个自己真正做深的点，用问题、方案、权衡、结果把深度打出来。


详细解释：


- 比如抓一个“协议解析重构”“日志系统优化”“Qt6 迁移兼容”展开。
- 深度不靠关键词堆砌，靠细节与取舍。


常见坑/追问：


- 一慌就开始泛泛而谈，会更像没做过。


### Q8: 🟢 怎样收尾才能让面试体验更好？


A: 结论：最后总结自己的匹配点，并表达对岗位方向的兴趣，让面试官更容易形成正向印象。


详细解释：


- 可简洁回扣：通信、Qt 桌面、Linux、工程化、稳定性。
- 若面试过程有没答好的点，也可补一句“之后我会再补这块”。


常见坑/追问：


- 不用长篇感言，简短真诚就行。
