## 一、Concepts 与约束

### Q1: ⭐🟢 concept 是解决什么问题的？


A: 结论：`concept` 用来给模板参数加“编译期约束”，让模板错误更早、更可读，也让接口语义更清晰。


详细解释：


- 传统模板出错常表现为一大串替换失败信息，不利于定位。
- `concept` 把“这个模板需要什么能力”直接写在接口上。
- 它是对 SFINAE/`enable_if` 的补强，不是简单语法糖。


代码示例：


```cpp
#include <concepts>

template <std::integral T>
T add(T a, T b) {
    return a + b;
}
```


常见坑/追问：


- `concept` 不会提升运行时性能，它主要提升可维护性和可诊断性。
- 追问：和 `static_assert` 的区别？前者更像接口约束，后者更像内部断言。

> 💡 **面试追问**：模板编译期展开有什么代价？如何减少模板实例化导致的代码膨胀？



### Q2: ⭐🟢 ranges 相比传统 STL 算法的优势是什么？


A: 结论：`ranges` 让算法与视图组合更自然，减少中间容器，提高表达力，并降低 iterator 边界错误。


详细解释：


- 传统 STL 常写 `std::sort(v.begin(), v.end())`。
- `ranges` 允许直接对 range 操作，还能配合 `views::filter/transform` 做管道式表达。
- 很适合“筛选 → 转换 → 聚合”这类数据流场景。


代码示例：


```cpp
#include <ranges>
#include <vector>
#include <iostream>

std::vector<int> v{1,2,3,4,5,6};
auto even = v | std::views::filter([](int x){ return x % 2 == 0; })
              | std::views::transform([](int x){ return x * 10; });
for (int x : even) std::cout << x << ' ';
```


常见坑/追问：


- `view` 往往是惰性求值，底层数据失效会出问题。
- 追问：为什么 `ranges` 更安全？因为算法直接接收 range，少写一对 begin/end。

> 💡 **面试追问**：vector 扩容时迭代器为何失效？如何用 `reserve` 优化？`std::deque` 和 `vector` 底层有何不同？



### Q3: ⭐🟡 coroutine 在 C++20 中是什么定位？


A: 结论：协程是一种语言级别的异步/挂起机制，适合表达状态机、异步 IO、生成器，但它本身不是调度器。


详细解释：


- 协程的核心是 `co_await`、`co_yield`、`co_return`。
- 它解决的是“异步代码写起来像同步代码”的问题。
- 真正怎么调度、在哪个线程恢复、怎么接入事件循环，要看框架实现，比如 Qt/Asio/自定义 executor。


代码示例：


```cpp
// 伪代码：展示语义，不是完整可编译实现
Task<int> foo() {
    int x = co_await async_read();
    co_return x + 1;
}
```


常见坑/追问：


- 协程不是线程，不会自动并行。
- 追问：Qt 里怎么用？通常要和事件循环、网络异步框架或第三方库配合。

> 💡 **面试追问**：线程池的核心参数如何调优？线程数设多少合适？



### Q4: ⭐🟡 std::span 适合什么场景？


A: 结论：`std::span` 是一个对连续内存的非拥有视图，适合替代“裸指针 + 长度”接口。


详细解释：


- 它不管理内存，只借用一段连续区域。
- 可统一接 `array`、`vector`、C 数组。
- 很适合序列化、协议解析、图像/音频 buffer 处理。


代码示例：


```cpp
#include <span>
#include <vector>

void process(std::span<const uint8_t> data) {
    // 只读处理
}

std::vector<uint8_t> buf{1,2,3};
process(buf);
```


常见坑/追问：


- `span` 不拥有数据，源对象销毁后它就悬空了。
- 追问：和 `string_view` 类似吗？类似，都是非拥有视图。

## 二、Ranges 与视图

### Q5: 🟡 std::jthread 比 std::thread 强在哪里？


A: 结论：`std::jthread` 自带 RAII join 和停止令牌，减少线程忘记回收或粗暴终止的问题。


详细解释：


- `std::thread` 析构时若仍 joinable 会 `std::terminate`。
- `jthread` 析构时会自动请求停止并 join，更符合资源管理思想。
- 它特别适合后台工作线程、轮询线程。


代码示例：


```cpp
#include <thread>
#include <chrono>

std::jthread worker([](std::stop_token st){
    while (!st.stop_requested()) {
        std::this_thread::sleep_for(std::chrono::milliseconds(100));
    }
});
```


常见坑/追问：


- `stop_token` 是协作式停止，不是强杀。
- 追问：为什么 Qt 线程常不用它？因为很多 Qt 项目已建立在 `QThread` 体系上。

> 💡 **面试追问**：与 GC 相比 RAII 的优势是什么？异常抛出时 RAII 为何仍然可靠？



### Q6: 🟡 三路比较运算符 &lt;=&gt; 有什么价值？


A: 结论：`&lt;=&gt;` 可以统一生成比较逻辑，减少重复实现 `&lt;`、`==`、`!=` 等运算符的样板代码。


详细解释：


- 它支持 strong/weak/partial ordering。
- 对值对象、配置结构体、排序键对象很有用。
- 编译器可基于它合成部分比较运算。


代码示例：


```cpp
#include <compare>

struct Point {
    int x;
    int y;
    auto operator<=>(const Point&) const = default;
};
```


常见坑/追问：


- 浮点比较常是 `partial_ordering`，不能简单等同于整数比较。

> 💡 **面试追问**：std::sort 使用什么算法？为什么不用归并排序？什么时候用 stable_sort？



### Q7: 🔴 module 能解决头文件什么问题？为什么现实项目里普及慢？


A: 结论：模块旨在减少头文件重复解析、降低宏污染、改善构建性能，但生态、编译器、构建系统和三方库兼容性让落地较慢。


详细解释：


- 传统头文件模型依赖文本包含，编译成本高，容易有宏副作用。
- module 更接近“编译后接口单元”，能提高大项目构建效率。
- 但现有工程大量基于头文件、模板库、条件编译，迁移成本不低。


代码示例：


```cpp
export module math;
export int add(int a, int b) { return a + b; }
```


常见坑/追问：


- 不同编译器对模块支持成熟度不同，CMake/toolchain 配合也容易踩坑。

## 三、协程

### Q8: 🔴 constexpr/consteval/constinit 应该怎么区分？


A: 结论：`constexpr` 表示“可在编译期求值”，`consteval` 表示“必须编译期求值”，`constinit` 表示“静态存储期对象必须常量初始化”。


详细解释：


- `constexpr` 函数可运行期也可编译期执行。
- `consteval` 常用于生成编译期校验或元编程结果。
- `constinit` 常用于避免静态初始化顺序问题。


代码示例：


```cpp
constexpr int f(int x) { return x * 2; }
consteval int g(int x) { return x + 1; }
constinit int global = 42;
```


常见坑/追问：


- `constinit` 不等于 `const`，它只保证初始化时机，不保证只读。

### Q9: ⭐🟡 C++20 的 `std::span` 解决了什么问题？


A: 结论：`std::span` 是对连续内存的非拥有视图，统一了"传数组/vector/裸指针+长度"的接口，消除越界风险，避免不必要的拷贝。


详细解释：


- 传统 API 常写 `void f(int* p, size_t n)`，容易出现长度不匹配。
- `std::span` 自动携带长度，编译器可做越界检查（debug 模式）。
- 支持 `std::span<const T>` 区分只读视图，语义更清晰。
- 可以从 C 数组、`std::vector`、`std::array` 等无缝构造。


代码示例：


```cpp
#include <span>
#include <vector>

void process(std::span<int> data) {
    for (auto& v : data) v *= 2;
}

int main() {
    std::vector<int> v{1, 2, 3};
    process(v);
    int arr[] = {4, 5, 6};
    process(arr);
}
```


常见坑/追问：


- `span` 是非拥有视图，所指内存的生命周期需调用方保证。
- 追问：和 `std::string_view` 的设计理念一致，都是"廉价视图"模式。

> 💡 **面试追问**：vector 扩容时迭代器为何失效？如何用 `reserve` 优化？`std::deque` 和 `vector` 底层有何不同？



### Q10: ⭐🟡 C++20 协程（coroutine）的核心机制是什么？


A: 结论：C++20 协程是可暂停/恢复的函数，通过 `co_await`/`co_yield`/`co_return` 驱动，底层由编译器生成状态机和 coroutine frame。


详细解释：


- 协程函数体会被编译器转换为状态机，挂起点保存局部变量到堆上的 coroutine frame。
- `co_await expr` 暂停当前协程，等待 awaitable 完成后恢复。
- `co_yield value` 产出一个值并挂起，适合实现 generator。
- `co_return` 结束协程，触发 promise 的 `return_value/return_void`。
- 标准库只提供机制，不提供 `Task`/`Generator` 等高级抽象（需要自己实现或用第三方库）。


代码示例：


```cpp
#include <coroutine>
#include <iostream>

struct Generator {
    struct promise_type {
        int current;
        auto get_return_object() { return Generator{std::coroutine_handle<promise_type>::from_promise(*this)}; }
        auto initial_suspend() { return std::suspend_always{}; }
        auto final_suspend() noexcept { return std::suspend_always{}; }
        auto yield_value(int v) { current = v; return std::suspend_always{}; }
        void return_void() {}
        void unhandled_exception() {}
    };
    std::coroutine_handle<promise_type> handle;
    bool next() { handle.resume(); return !handle.done(); }
    int value() { return handle.promise().current; }
    ~Generator() { if (handle) handle.destroy(); }
};

Generator count(int n) {
    for (int i = 0; i < n; ++i) co_yield i;
}
```


常见坑/追问：


- coroutine frame 默认堆分配，编译器可能优化消除（HALO）。
- 追问：协程与线程的区别——协程是协作式调度，线程是抢占式。

## 四、其他新特性

### Q11: 🟡 `std::jthread` 和 `std::thread` 有什么区别？


A: 结论：`std::jthread` 是"joining thread"，析构时自动 join，并内置 `stop_token` 支持协作式取消，避免 `std::thread` 析构时因未 join 而 `std::terminate`。


详细解释：


- `std::thread` 析构时若未 join 或 detach，程序直接终止。
- `std::jthread` 析构时自动调用 `request_stop()` + `join()`。
- `stop_token` 提供标准的取消机制，线程函数通过轮询 `stop_token::stop_requested()` 响应。


代码示例：


```cpp
#include <thread>
#include <stop_token>
#include <iostream>

void worker(std::stop_token st) {
    while (!st.stop_requested()) {
        std::cout << "working...\n";
        std::this_thread::sleep_for(std::chrono::milliseconds(100));
    }
}

int main() {
    std::jthread t(worker); // 析构时自动 stop + join
}
```


常见坑/追问：


- `stop_token` 是协作式取消，线程自己需要检查，不是强制终止。
- 追问：和 `pthread_cancel` 的本质区别？后者是抢占式，前者需要线程配合。

### Q12: 🟡 C++20 三路比较运算符 `<=>` 怎么使用？


A: 结论：`<=>` 是太空船运算符，返回比较类别对象，通过一次定义可自动生成所有关系运算符，消除大量样板代码。


详细解释：


- 返回类型有 `strong_ordering`（整数）、`weak_ordering`（忽略某些差异）、`partial_ordering`（浮点）。
- 定义 `operator<=>` 后，编译器会自动合成 `<、>、<=、>=`。
- 同时定义 `==` 可得到全套六个比较运算符。
- 用 `= default` 可按成员字典序自动生成。


代码示例：


```cpp
#include <compare>

struct Point {
    int x, y;
    auto operator<=>(const Point&) const = default; // 自动生成全套
};

Point a{1, 2}, b{1, 3};
bool lt = a < b;   // true
bool eq = a == b;  // false
```


常见坑/追问：


- 浮点数返回 `partial_ordering`，因为 NaN 无法比较。
- 追问：`= default` 的三路比较是按成员声明顺序字典序比较的。

### Q13: ⭐🟡 `std::format` 相比 `printf`/`std::stringstream` 的优势是什么？


A: 结论：`std::format` 提供类型安全、可扩展、高性能的字符串格式化，编译期检查格式字符串，无 `printf` 的类型不安全风险。


详细解释：


- `printf` 类型安全由程序员保证，格式串错误是 UB。
- `std::stringstream` 冗长且慢。
- `std::format` 使用 `{}` 占位符，支持宽度、精度、填充等丰富格式选项。
- 可通过特化 `std::formatter` 支持自定义类型。
- C++23 的 `std::print` 直接写到输出，更方便。


代码示例：


```cpp
#include <format>
#include <string>
#include <iostream>

int main() {
    std::string s = std::format("Hello, {}! Pi ≈ {:.2f}", "world", 3.14159);
    std::cout << s << '\n';
    // 宽度对齐
    std::cout << std::format("{:>10}", "right") << '\n';
}
```


常见坑/追问：


- C++20 `std::format` 编译支持情况：GCC 13+、Clang 14+、MSVC 2019 16.10+。
- 追问：如何让自定义类型支持 `std::format`？特化 `std::formatter<T>`。

### Q14: 🟡 C++20 modules 解决了什么问题？和头文件有何不同？


A: 结论：modules 替代了 `#include` 的文本包含模型，提供更快的编译、更好的隔离性，消除宏污染和重复解析开销。


详细解释：


- 传统 `#include` 每次包含都重新解析头文件，大型项目编译极慢。
- modules 编译后生成 BMI（Binary Module Interface），后续使用直接读取，大幅提速。
- 模块内的宏不外泄，避免宏污染全局命名空间。
- `export` 关键字精确控制对外接口。
- 目前工具链支持（CMake、IDE）仍在完善中，迁移需谨慎。


代码示例：


```cpp
// math.cppm
export module math;

export int add(int a, int b) { return a + b; }
int helper() { return 0; } // 不导出，模块私有

// main.cpp
import math;
int main() { return add(1, 2); }
```


常见坑/追问：


- `import std;` 可以在 C++23 中直接导入标准库，编译极快。
- 追问：PCH（预编译头）和 modules 的区别——PCH 是对现有模型的加速，modules 是替换模型。

### Q15: ⭐🔴 C++20 有哪些对并发编程的新支持？


A: 结论：C++20 在并发方面新增了 `std::latch`、`std::barrier`、`std::counting_semaphore`、`std::atomic_ref`、`atomic::wait/notify`，以及协程对异步的支持，覆盖了更多同步原语场景。


详细解释：


- `std::latch`：一次性计数器，等所有线程都报到后继续，不可复用。
- `std::barrier`：可复用的屏障，所有线程到齐后执行 completion function，再重置。
- `std::counting_semaphore`：信号量，经典同步原语，`std::binary_semaphore` 是特化。
- `std::atomic_ref<T>`：对非原子对象进行原子操作，无需修改对象类型本身。
- `atomic::wait()`：高效等待原子变量变化，避免忙等。


代码示例：


```cpp
#include <latch>
#include <thread>
#include <vector>
#include <iostream>

int main() {
    std::latch done{3};
    std::vector<std::thread> workers;
    for (int i = 0; i < 3; ++i) {
        workers.emplace_back([&, i] {
            std::cout << "worker " << i << " done\n";
            done.count_down();
        });
    }
    done.wait(); // 等所有 worker 完成
    for (auto& t : workers) t.join();
}
```


常见坑/追问：


- `latch` 不可复用，`barrier` 可复用，选型要注意。
- 追问：`atomic::wait()` 内部用 futex（Linux）或类似机制，比自旋更省 CPU。
