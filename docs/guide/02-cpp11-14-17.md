# 2. C++11/14/17 新特性

### Q1: ⭐🟢 为什么 C++11 被称为现代 C++ 的起点？


A: 结论：因为 C++11 引入了移动语义、智能指针、lambda、线程库、auto、nullptr 等一整套新范式，直接改变了写代码的习惯。很多今天习以为常的写法，都是从 C++11 开始普及的。


详细解释：


- 资源管理从“手写 new/delete”逐步转向 RAII + 智能指针。
- 泛型代码从 functor 转向 lambda。
- 并发从平台 API 转向标准线程库。
- 接口表达更偏语义化，而不是纯技巧化。


代码示例：


```cpp
auto f = [](int a, int b) { return a + b; };
std::unique_ptr<int> p(new int(42));
```


常见坑/追问：


- 面试时别只列清单，最好能说出“这些特性改变了什么开发方式”。


### Q2: ⭐🟢 C++11 最常考的新特性有哪些？


A: 结论：最常考的是右值引用、移动语义、智能指针、lambda、auto、nullptr、范围 for、线程库和 `constexpr`。这些基本是现代 C++ 面试必答题。


详细解释：


- 右值引用/移动语义：减少深拷贝。
- 智能指针：表达所有权。
- lambda：就地定义可调用对象。
- `auto`：简化类型书写。
- `nullptr`：解决空指针重载歧义。
- 标准线程库：统一并发接口。


代码示例：


```cpp
for (auto& x : vec) {
    x *= 2;
}
```


常见坑/追问：


- `auto` 不会“动态推断”，它是编译期类型推导。


### Q3: 🟡 C++14 相比 C++11 的定位是什么？


A: 结论：C++14 更像是 C++11 的增强补丁版，不是革命性变化，但大幅提升了可用性和书写体验。它解决了很多“C++11 能做但写起来别扭”的问题。


详细解释：


- 引入 `std::make_unique`。
- 泛型 lambda 更自然。
- 返回类型推导放宽。
- `constexpr` 限制减少。


代码示例：


```cpp
auto add = [](auto a, auto b) { return a + b; };
auto p = std::make_unique<int>(10);
```


常见坑/追问：


- 面试官可能问：为什么 `make_unique` 在 C++11 没有？本质是标准演进节奏问题。


### Q4: ⭐🟡 C++17 最实用的改进有哪些？


A: 结论：C++17 最常落地的是结构化绑定、`if constexpr`、`std::optional`、`std::variant`、`std::string_view` 和文件系统库。它让现代 C++ 写法更完整、更像一门成熟工程语言。


详细解释：


- 结构化绑定提升可读性。
- `if constexpr` 改善模板分支。
- `optional/variant` 提升类型表达力。
- `string_view` 减少不必要拷贝。
- `&lt;filesystem&gt;` 提供统一文件系统 API。


代码示例：


```cpp
if (auto it = m.find(key); it != m.end()) {
    auto [k, v] = *it;
}
```


常见坑/追问：


- `string_view` 是非拥有视图，生命周期是高频坑。


### Q5: ⭐🟢 nullptr 为什么比 NULL 更好？


A: 结论：`nullptr` 有独立类型 `std::nullptr_t`，能准确表达“空指针”，避免整数 0 带来的重载歧义。现代 C++ 里应优先使用 `nullptr`。


详细解释：


- `NULL` 本质上很多实现就是 0。
- 传给重载函数时可能匹配到整型版本。
- `nullptr` 则只表示空指针语义。


代码示例：


```cpp
void f(int);
void f(char*);

// f(NULL);   // 可能走 int
f(nullptr);   // 明确走 char*
```


常见坑/追问：


- `nullptr` 不是关键字替代 `0`，而是语义更强的专用空指针字面量。


### Q6: 🟡 auto 的优点和边界是什么？


A: 结论：`auto` 能显著减少模板和复杂类型书写负担，提高可读性；但如果把关键语义藏掉，也会让代码更难懂。它该用在“类型明显但书写冗长”的地方。


详细解释：


- 适合迭代器、lambda、复杂模板返回值。
- 不适合隐藏精度、所有权、值/引用语义这些关键信息。
- `auto` 推导时会丢顶层 `const`，引用语义也要显式写 `auto&amp;`/`const auto&amp;`。


代码示例：


```cpp
auto x = 42;          // int
const auto& ref = x;  // 引用语义保留
```


常见坑/追问：


- `auto` 不是“偷懒”，要看是否让代码更清楚。


### Q7: ⭐🟡 std::optional 解决了什么问题？


A: 结论：`std::optional&lt;T&gt;` 用类型系统表达“这个值可能不存在”，比返回特殊值或裸指针更安全、可读性更好。它非常适合解析、查找、配置读取等场景。


详细解释：


- 避免用 `-1`、空串、魔法值表达失败。
- 调用方必须显式检查是否有值。
- 有助于 API 语义更清晰。


代码示例：


```cpp
std::optional<int> parseInt(const std::string& s);

if (auto v = parseInt("123")) {
    std::cout << *v << '\n';
}
```


常见坑/追问：


- `optional&lt;T&amp;&gt;` 不存在，引用场景要用指针或 `std::reference_wrapper`。


### Q8: 🟡 std::variant 和 std::any 有什么区别？


A: 结论：`variant` 是受约束的类型安全联合体，备选类型在编译期确定；`any` 则更灵活，能装任意类型，但类型信息检查发生在运行期。能用 `variant` 时，通常优先 `variant`。


详细解释：


- `variant` 更适合有限状态或多种业务消息类型。
- `any` 更像“类型擦除容器”，灵活但约束弱。
- `variant` 的可维护性、可分析性一般更好。


代码示例：


```cpp
std::variant<int, std::string> v = "hello";
std::visit([](const auto& x) { std::cout << x; }, v);
```


常见坑/追问：


- `std::get` 取错类型会抛异常，`std::get_if` 更稳。


### Q9: ⭐🟡 if constexpr 和普通 if 的本质区别是什么？


A: 结论：`if constexpr` 是编译期分支，未命中的分支不会实例化；普通 `if` 是运行期分支，两边代码都必须先通过编译。它在模板编程里特别重要。


详细解释：


- 模板中常需要针对不同类型走不同逻辑。
- 普通 `if` 即使条件永远为假，语法也必须合法。
- `if constexpr` 可以在编译阶段剪掉无效分支。


代码示例：


```cpp
template <typename T>
void print(const T& v) {
    if constexpr (std::is_integral_v<T>) {
        std::cout << "integral\n";
    } else {
        std::cout << "other\n";
    }
}
```


常见坑/追问：


- `if constexpr` 不是宏替换，它仍然受正常作用域和语义规则约束。


### Q10: 🟡 std::string_view 的优点和风险是什么？


A: 结论：`string_view` 的最大优点是零拷贝传参，但它不拥有底层数据，因此生命周期风险是最大坑。它适合只读视图，不适合持久保存陌生来源的数据。


详细解释：


- 参数传递时可避免构造临时 `std::string`。
- 与字符串字面量、已有字符串缓冲区配合很方便。
- 一旦源字符串销毁，`string_view` 就会悬空。


代码示例：


```cpp
void print(std::string_view sv) {
    std::cout << sv << '\n';
}
```


常见坑/追问：


- 不要返回指向局部字符串的 `string_view`。


### Q11: ⭐🟢 范围 for、lambda、智能指针为什么常被视为“现代 C++ 基础素养”？


A: 结论：因为这三者直接对应现代 C++ 的三大日常能力：容器遍历、局部行为抽象、资源所有权管理。不会它们，很难写出今天主流风格的 C++。


详细解释：


- 范围 for 让遍历更直接。
- lambda 让短逻辑贴近调用点。
- 智能指针让资源边界更明确。


代码示例：


```cpp
std::vector<int> v{1,2,3};
std::for_each(v.begin(), v.end(), [](int x){ std::cout << x << ' '; });
```


常见坑/追问：


- lambda 捕获方式、引用失效、共享所有权滥用都是高频追问。


### Q12: 🟢 工程上 C++11/14/17 一般怎么选？


A: 结论：如果工具链允许，新项目至少选 C++17；老项目受编译器、依赖链限制时可以停在 C++14；除非环境特别旧，一般不建议新项目只用 C++11。因为 C++17 的收益已经很大且相对稳定。


详细解释：


- C++17 提供更成熟的语言和标准库支持。
- 现代库生态通常也更偏向 C++17 起步。
- 决策时还要考虑 ABI、第三方依赖和 CI 环境。


常见坑/追问：


- 不要脱离团队环境空谈“越新越好”，工具链兼容性也很关键。


### Q13: ⭐🟡 std::move 和 std::forward 有什么区别？


A: 结论：`std::move` 无条件将左值转成右值引用，触发移动语义；`std::forward` 是条件转发，只有当模板参数推断为右值时才转成右值，用于完美转发。二者都只是强制类型转换，本身不做任何移动或拷贝。


详细解释：


- `std::move(x)` 等价于 `static_cast<T&&>(x)`，让资源被"偷走"。
- `std::forward<T>(x)` 保留传入时的值类别（左值/右值），适合泛型转发。
- 在非模板代码中几乎只用 `std::move`；在模板参数包、完美转发场景才用 `std::forward`。


代码示例（如有）：


```cpp
template<typename T>
void wrapper(T&& arg) {
    // 完美转发：保留原始值类别
    target(std::forward<T>(arg));
}

std::string s = "hello";
std::string s2 = std::move(s); // s 之后为空
```


常见坑/追问：


- `std::move` 之后原对象处于"有效但未指定"状态，不要再用它的值。
- `std::forward` 只有在模板推导上下文中才有意义，单独用在普通函数里没有效果。


### Q14: ⭐🟡 什么是 SFINAE？enable_if 怎么用？


A: 结论：SFINAE（Substitution Failure Is Not An Error）是 C++ 模板替换失败不报错、只是从候选集中排除的规则。`std::enable_if` 利用 SFINAE 根据条件启用或禁用模板重载，是实现编译期类型约束的经典方式。C++17 的 `if constexpr` 和 C++20 的 concept 是更现代的替代。


详细解释：


- 模板实例化时若类型替换产生无效表达式，该重载被静默丢弃，而不是报错。
- `enable_if<cond, T>::type` 仅在 `cond == true` 时存在，否则替换失败。
- 常用于约束只接受整数、指针、具体接口的模板函数。


代码示例（如有）：


```cpp
#include <type_traits>

// 只对整型有效
template<typename T, typename = std::enable_if_t<std::is_integral_v<T>>>
void process(T val) {
    // ...
}

// C++17 更清晰：
template<typename T>
void process2(T val) {
    if constexpr (std::is_integral_v<T>) { /* ... */ }
}
```


常见坑/追问：


- `enable_if` 出错信息极难读；新项目优先用 `if constexpr` 或 concept（C++20）。
- 注意 `enable_if` 写在返回值还是模板参数里，两种写法各有适用场景。


### Q15: 🟡 C++17 的结构化绑定（structured bindings）怎么用？


A: 结论：结构化绑定让你用 `auto [a, b] = ...` 的方式直接解包 pair、tuple、struct 甚至数组，代替手动 `.first/.second` 或 `get<N>`，代码更简洁直观。


详细解释：


- 支持 `std::pair`、`std::tuple`、聚合类型（public 成员）、C 数组。
- 编译器生成绑定变量，底层仍是引用原始对象成员，不一定做拷贝。
- 结合范围 for 遍历 map 非常方便。


代码示例（如有）：


```cpp
#include <map>
#include <tuple>

// map 遍历
std::map<int, std::string> m{{1, "one"}, {2, "two"}};
for (auto& [key, val] : m) {
    std::cout << key << " = " << val << '\n';
}

// tuple 解包
auto [x, y, z] = std::make_tuple(1, 2.0, "hi");

// 函数返回多值
std::pair<int,bool> lookup(int id);
auto [value, found] = lookup(42);
```


常见坑/追问：


- 绑定的变量本质是引用，修改 `val` 会修改 map 的值（加 `const` 可防止）。
- 不能只绑部分成员，必须全部解包。
