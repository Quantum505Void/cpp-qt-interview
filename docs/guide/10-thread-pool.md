# 10. 线程池


↑ 回到目录


### Q1: ⭐🟢 为什么需要线程池，而不是每次都创建线程？


A: 结论：因为线程创建和销毁有成本，线程数失控会拖垮系统，而线程池能复用工作线程、统一调度任务并控制并发度。线程池本质上是在“节流”和“复用”。


详细解释：


- 新建线程涉及栈空间、内核对象、调度开销。
- 短任务频繁创建线程尤其浪费。
- 线程池还能作为系统并发闸门。


常见坑/追问：


- 线程池不是让程序自动更快，它是让并发更可控。


### Q2: ⭐🟡 线程池的核心组成有哪些？


A: 结论：一个基础线程池通常包含工作线程集合、任务队列、同步机制、停止标志和任务提交接口。设计得更完整时，还会有队列上限、拒绝策略、监控统计和异常处理。


详细解释：


- worker 负责循环取任务执行。
- queue 负责缓存待执行任务。
- mutex/cv 负责同步。
- stop flag 控制优雅退出。


常见坑/追问：


- 面试官常追问：队列满了怎么办？这就是线程池设计成熟度的重要体现。


### Q3: ⭐🟡 线程池任务执行时为什么要尽量在锁外运行？


A: 结论：因为锁内只应做取任务这类最小操作，如果把任务执行放在锁内，会把整个线程池串行化，严重增加锁竞争和延迟。锁是守门员，不是陪跑员。


详细解释：


- 正确模式通常是：加锁取任务 -> 解锁 -> 执行任务。
- 这样其他 worker 才能并发取任务。


代码示例：


```cpp
std::function<void()> task;
{
    std::unique_lock<std::mutex> lock(m);
    task = std::move(tasks.front());
    tasks.pop();
}
task();
```


常见坑/追问：


- 持锁执行任务还容易导致回调重入和隐藏死锁。


### Q4: ⭐🟡 std::packaged_task、future 在线程池里有什么用？


A: 结论：它们用于把异步任务结果安全地返回给调用方，并顺带承载异常传播。相比只提交 `void()` 任务，它们让线程池更像一个异步计算框架。


详细解释：


- `packaged_task` 包装可调用对象。
- 提交后可拿到 `future`。
- 调用方用 `get()` 获取结果或异常。


代码示例：


```cpp
auto fut = std::async(std::launch::async, [] { return 7; });
int x = fut.get();
```


常见坑/追问：


- `future.get()` 只能调用一次，接口设计时要注意消费模型。


### Q5: ⭐🟡 线程池线程数一般怎么估？


A: 结论：CPU 密集型任务线程数通常接近 CPU 核数，IO 密集型可以更多，但最终要靠压测和监控校准。没有一个万能常数，只有与业务负载相匹配的经验值。


详细解释：


- CPU 密集：过多线程只会增加切换成本。
- IO 密集：线程大部分时间在等待，可适当放大。
- 混合负载场景经常要分池。


常见坑/追问：


- 面试时能提“区分 CPU 池和 IO 池”会比较加分。


### Q6: 🟡 线程池适合哪些场景？


A: 结论：线程池适合大量短到中等时长的异步任务，如日志处理、文件扫描、请求处理、后台计算、批量任务调度等。它不适合所有并发场景，但非常适合作为通用执行器。


详细解释：


- 任务之间相对独立。
- 需要统一控制并发度。
- 任务提交频繁但单个任务不值得专门创建线程。


常见坑/追问：


- “一个超长任务永久占坑”会影响池内其他任务调度，这是设计时要考虑的。


### Q7: ⭐🟡 什么场景不适合线程池？


A: 结论：超长生命周期任务、强线程亲和任务、必须独占线程上下文的任务，以及无法合理切分的串行流程，都不一定适合线程池。不要把线程池当作并发万能胶。


详细解释：


- 比如 GUI 主线程任务、特殊实时线程、专用网络事件循环线程。
- 线程池强调共享 worker，而不是固定线程身份。


常见坑/追问：


- 如果任务依赖 TLS、线程本地状态或特定线程初始化，普通线程池可能不合适。


### Q8: ⭐🟡 线程池里的异常应该怎么处理？


A: 结论：不要让异常直接逃出工作线程边界，否则会触发 `std::terminate` 或让线程悄悄崩掉。正确做法是在线程边界捕获异常，再通过 future、回调或日志上报。


详细解释：


- 线程函数是异常边界。
- 异常处理策略要统一，不能每个任务各自乱飞。
- 对业务线程池来说，异常可观测性非常重要。


常见坑/追问：


- “线上线程池越来越少”有时就是某个任务抛异常把 worker 干掉了。


### Q9: ⭐🟡 队列满了怎么办？线程池要有哪些拒绝策略？


A: 结论：常见策略包括阻塞提交、直接拒绝、丢弃旧任务、丢弃新任务、调用者线程自己执行。具体选哪种，取决于业务是否允许降级、是否强调延迟还是成功率。


详细解释：


- 阻塞提交：稳，但可能把上游拖住。
- 直接拒绝：边界清晰，适合保护系统。
- 调用者执行：一种自然背压方式。


常见坑/追问：


- 没有上限和拒绝策略的线程池，本质上只是把问题换成内存堆积。


### Q10: ⭐🟡 线程池如何优雅关闭？


A: 结论：优雅关闭的步骤通常是：停止接收新任务、唤醒所有 worker、处理剩余任务或按策略丢弃，然后 join 线程。关闭流程设计不好，线程池很容易在程序退出时卡死。


详细解释：


- stop flag 要受锁保护或用原子变量。
- `condition_variable` 要被 notify_all。
- join 前确保线程能从 wait 中醒来并看到退出条件。


常见坑/追问：


- 析构里忘记唤醒等待线程，是线程池退出死锁高频坑。


### Q11: ⭐🟡 Qt 中 QThreadPool 和 QtConcurrent 怎么选？


A: 结论：需要更底层控制任务对象、线程池参数和复用策略时用 `QThreadPool`；想快速把函数异步跑起来并拿结果时，用 `QtConcurrent` 更方便。前者偏执行器，后者偏便捷并发接口。


详细解释：


- `QThreadPool` 常搭配 `QRunnable`。
- `QtConcurrent` 更适合 map/filter/run 等函数式任务。
- 两者底层有联系，但抽象层级不同。


代码示例：


```cpp
auto future = QtConcurrent::run([] {
    return 42;
});
```


常见坑/追问：


- QtConcurrent 很方便，但复杂调度控制和资源治理能力不如自己管理线程池灵活。


### Q12: ⭐🟡 面试里怎么评价一个线程池设计是否成熟？


A: 结论：可以从这几个维度看：是否支持返回值与异常、是否有队列上限与拒绝策略、是否优雅关闭、锁竞争是否合理、是否能监控任务积压和耗时。真正成熟的线程池，关心的不只是“跑起来”，而是“稳定地跑很久”。


详细解释：


- 线程池是基础设施，可靠性比 API 花哨更重要。
- 监控和背压能力往往是区分玩具实现和工程实现的关键。


常见坑/追问：


- 如果面试官追问，你可以补一句：CPU 池/IO 池分离通常也是成熟度体现。


### Q13: ⭐🟡 如何向线程池提交带返回值的任务？


A: 结论：用 `std::packaged_task` 包装可调用对象，绑定到 `std::future`，将 task 投入队列，调用方通过 future 阻塞等待结果。这是 C++ 标准库实现"异步计算 + 结果回收"的惯用模式。


详细解释：


- `packaged_task` 内部自动管理 promise/future 对，调用任务时自动 `set_value`。
- 模板 `submit` 函数返回 `std::future<ReturnType>`，调用方可以选择在合适时机 `get()`。
- 注意 `packaged_task` 不可拷贝，需用 `shared_ptr` 或 `std::move` 放入队列。


代码示例（如有）：


```cpp
template<typename F, typename... Args>
auto submit(F&& f, Args&&... args) -> std::future<std::invoke_result_t<F, Args...>> {
    using RetT = std::invoke_result_t<F, Args...>;
    auto task = std::make_shared<std::packaged_task<RetT()>>(
        std::bind(std::forward<F>(f), std::forward<Args>(args)...)
    );
    std::future<RetT> fut = task->get_future();
    {
        std::lock_guard<std::mutex> lk(mtx_);
        tasks_.push([task]{ (*task)(); });
    }
    cv_.notify_one();
    return fut;
}
```


常见坑/追问：


- `future.get()` 会重新抛出任务内的异常，调用方要做好异常处理。
- 如果线程池已关闭，向队列 push 后应返回无效 future 或抛异常，而不是静默丢失任务。


### Q14: 🟡 线程池中工作线程数应该怎么确定？


A: 结论：CPU 密集型任务设为 CPU 核心数（`std::thread::hardware_concurrency()`）；IO 密集型任务因为等待比例高，可以设为核心数的 2~4 倍；混合型任务需要基准测试。没有万能公式，监控 + 调参才是正解。


详细解释：


- CPU 密集：更多线程只增加上下文切换，超过核心数收益递减。
- IO 密集：线程等 IO 时不占 CPU，可以更多线程重叠等待，提升吞吐。
- Little's Law 视角：平均吞吐 = 并发数 / 平均延迟，线程数要匹配目标吞吐与延迟。
- 动态调整：高级线程池（如 Java ForkJoinPool）会根据任务阻塞率动态补充线程。


常见坑/追问：


- `hardware_concurrency()` 可能返回 0（失败时），代码要做保底处理。
- 线程数太少，任务积压；太多，内存和栈空间浪费，调度开销增大。


### Q15: ⭐🟡 工作窃取（work stealing）线程池和普通线程池有什么区别？


A: 结论：普通线程池共享一个全局队列，所有线程竞争同一把锁；工作窃取线程池每个线程有自己的本地双端队列，线程优先消费自己的任务，空闲时从其他线程队列的"尾部"偷取任务，减少锁竞争，提升局部性。


详细解释：


- 本地队列用 `deque`，线程从前端 push/pop（LIFO，利用缓存局部性）。
- 偷取从其他线程队列尾端 pop（FIFO），减少碰撞概率。
- 适合 fork-join 计算密集型负载（如递归分治），Cilk、Intel TBB、Java ForkJoinPool 都用此模型。
- 缺点：实现复杂，需要无锁或细粒度锁的 deque；对 IO 密集型任务收益不明显。


代码示例（如有）：


```cpp
// 概念示意：每个线程持有自己的 deque
// push_front / pop_front  <-- 线程本地操作（LIFO）
// pop_back                <-- 其他线程窃取操作（FIFO）
// 全局窃取逻辑：
// if (local_deque.empty()) {
//     steal from random other thread's deque.back();
// }
```


常见坑/追问：


- 工作窃取不是银弹，任务粒度太细时窃取开销可能超过收益。
- 追问：TBB（Threading Building Blocks）和 std::thread 线程池的区别？TBB 内置工作窃取，适合数据并行；std::thread 池更通用灵活。
