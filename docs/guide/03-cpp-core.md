## 一、面向对象基础

### Q1: ⭐🟢 OOP 三大特性在 C++ 里怎么理解？


A: 结论：封装是隐藏实现细节，继承是建立 is-a 关系，多态是统一接口下表现出不同行为。面试时最好别背定义，要能联系实际设计。


详细解释：


- 封装强调边界与职责。
- 继承强调接口/实现复用，但不能滥用。
- 多态强调通过抽象接口屏蔽具体实现差异。


常见坑/追问：


- 继承不是复用代码的唯一方式，很多场景组合优于继承。

### Q2: ⭐🟢 RAII 到底是什么？为什么它是 C++ 的灵魂？


A: 结论：RAII 是“资源获取即初始化”，把资源生命周期绑定到对象生命周期。它是 C++ 管理内存、文件、锁、句柄最核心的思想。


详细解释：


- 构造时获取资源，析构时释放资源。
- 即使函数中途 return 或抛异常，也能自动释放。
- 智能指针、锁守卫、本质上都在贯彻 RAII。


代码示例：


```cpp
std::mutex m;
void f() {
    std::lock_guard<std::mutex> lock(m);
    // 临界区
}
```


常见坑/追问：


- RAII 不只管内存，文件、socket、数据库连接都适用。

> 💡 **面试追问**：与 GC 相比 RAII 的优势是什么？异常抛出时 RAII 为何仍然可靠？



### Q3: 🟢 C++ 的内存区域通常怎么划分？


A: 结论：面试里常见回答是栈、堆、静态/全局区、常量区、代码区。虽然不同平台实现细节不同，但这个划分足够回答大多数基础题。


详细解释：


- 栈：自动分配释放，适合局部变量。
- 堆：动态分配，生命周期灵活。
- 静态/全局区：程序整个运行期存在。
- 常量区：字符串字面量、只读常量等。
- 代码区：机器指令。


常见坑/追问：


- “在栈上/堆上”是对象存储位置，不是类型属性。

> 💡 **面试追问**：内存泄漏如何定位？Valgrind 和 AddressSanitizer 各自适合什么场景？



### Q4: ⭐🟡 explicit 关键字解决了什么问题？


A: 结论：`explicit` 用来阻止不希望发生的隐式转换，避免构造函数或转换运算符被编译器偷偷调用。它是接口设计中非常高频的防坑工具。


详细解释：


- 单参数构造函数默认可能参与隐式转换。
- 这会让函数调用、重载解析出现意外行为。
- `explicit` 让调用者必须显式写出转换意图。


代码示例：


```cpp
class Size {
public:
    explicit Size(int v) : value(v) {}
private:
    int value;
};

void setSize(Size s);
// setSize(10); // 错误，避免了隐式转换
setSize(Size(10));
```


常见坑/追问：


- C++11 起 `explicit` 也可用于转换运算符。

### Q5: 🟡 volatile 在 C++ 里到底是干什么的？


A: 结论：`volatile` 表示对象的值可能在程序显式控制之外发生变化，因此编译器不应对相关读写做过度优化。它不是线程同步工具，也不能替代原子操作。


详细解释：


- 常见场景：内存映射寄存器、硬件状态位、信号处理标志。
- 它保证“每次都真的去读/写”，不保证原子性和顺序同步。
- 多线程共享变量应使用 `std::atomic` 或锁。


常见坑/追问：


- “volatile 保证线程安全”是典型错误答案。

> 💡 **面试追问**：线程池的核心参数如何调优？线程数设多少合适？




## 二、关键字与类型系统

### Q6: ⭐🟡 this 指针是什么？有哪些高频追问？


A: 结论：`this` 是成员函数中的隐式指针，指向当前调用该成员函数的对象。它本质上是编译器传入的隐藏参数。


详细解释：


- 在非静态成员函数中存在。
- `this` 的类型在 `const` 成员函数中会变成指向 const 的指针。
- 可用来实现链式调用、区分成员与参数同名、返回自身引用。


代码示例：


```cpp
class A {
public:
    A& setX(int x) {
        this->x = x;
        return *this;
    }
private:
    int x{};
};
```


常见坑/追问：


- 静态成员函数没有 `this`。
- 在构造/析构期间使用 `this` 逃逸到外部是危险设计。

### Q7: ⭐🟡 四种 cast 分别怎么用？


A: 结论：`static_cast` 用于常规显式转换，`dynamic_cast` 用于多态层级安全向下转型，`const_cast` 用于增加/移除 cv 限定，`reinterpret_cast` 用于底层比特级重解释。面试里要强调“能用窄的，不用野的”。


详细解释：


- `static_cast`：数值转换、上行转型、显式构造。
- `dynamic_cast`：运行期检查，需要多态基类。
- `const_cast`：修改 const/volatile 修饰。
- `reinterpret_cast`：最危险，只做底层解释，不保证语义安全。


代码示例：


```cpp
Base* b = new Derived;
Derived* d = dynamic_cast<Derived*>(b);
```


常见坑/追问：


- 用 `const_cast` 去改原本就是 const 的对象，行为未定义。
- `dynamic_cast` 失败时指针返回空，引用会抛 `std::bad_cast`。

### Q8: 🟡 friend 有什么用？为什么它既常见又有争议？


A: 结论：`friend` 允许外部函数或类访问当前类的私有/保护成员，用于紧密协作场景。它有用，但会打破封装边界，所以应少而准。


详细解释：


- 常用于运算符重载、测试辅助、工厂函数、强耦合协作类。
- 它不是继承，也不是双向关系。
- 被声明为 friend 的对象不属于成员。


代码示例：


```cpp
class A {
    friend std::ostream& operator<<(std::ostream& os, const A& a);
private:
    int x{42};
};
```


常见坑/追问：


- `friend` 应该是“有意识地开后门”，不是图省事乱用。

### Q9: ⭐🟡 运算符重载有哪些原则？


A: 结论：运算符重载要遵循直觉语义，不能为了炫技而破坏可读性。重载的价值在于让自定义类型行为像内建类型一样自然，而不是制造惊喜。


详细解释：


- `+` 应表达组合或加法，`==` 应表达等价关系。
- 修改对象状态的运算符要注意返回值语义。
- 输入输出运算符通常写成非成员友元函数。


代码示例：


```cpp
class Vec2 {
public:
    Vec2(int x, int y) : x(x), y(y) {}
    Vec2 operator+(const Vec2& other) const {
        return Vec2(x + other.x, y + other.y);
    }
private:
    int x, y;
};
```


常见坑/追问：


- 不要重载 `&amp;&amp;`、`,` 这类容易破坏直觉和求值规则的运算符。

### Q10: 🟢 struct 和 class 的区别是什么？


A: 结论：在 C++ 里两者本质几乎一样，核心区别只是默认访问权限和默认继承权限不同：`struct` 默认 public，`class` 默认 private。工程上更多是语义风格区别。


详细解释：


- 简单数据聚合、配置对象常用 `struct`。
- 强调封装、抽象、约束时常用 `class`。
- 功能上二者都能有成员函数、继承、模板等。


常见坑/追问：


- 别把 C 语言里的 struct 概念原样套到 C++。

## 三、函数与运算符

### Q11: ⭐🟡 inline 和宏有什么区别？


A: 结论：`inline` 是 C++ 语言级函数机制，宏只是预处理文本替换。现代 C++ 里应优先 `inline`、`constexpr`、模板，而不是函数式宏。


详细解释：


- `inline` 有类型检查、作用域、调试信息。
- 宏没有类型系统，容易重复求值。
- `inline` 也不等于编译器一定内联，它主要影响 ODR 和链接语义。


代码示例：


```cpp
#define SQR(x) ((x) * (x))
inline int sqr(int x) { return x * x; }
```


常见坑/追问：


- `SQR(i++)` 会出事，宏重复求值是经典坑。

### Q12: ⭐🟡 函数指针、std::function、Lambda 怎么比较？


A: 结论：函数指针轻量但能力弱，lambda 最自然，`std::function` 最通用但有一定类型擦除开销。选型原则是：先看语义表达，再考虑性能。


详细解释：


- 函数指针只能表示可退化成函数地址的调用目标。
- lambda 可捕获上下文，适合局部逻辑。
- `std::function` 可统一包装函数、lambda、bind、仿函数。


代码示例：


```cpp
void run(const std::function<void()>& fn) {
    fn();
}

int x = 10;
run([x] { std::cout << x << '\n'; });
```


常见坑/追问：


- 高性能热路径中，`std::function` 的分配和间接调用可能要关注。

### Q13: 🟡 constexpr 和 const 的区别是什么？


A: 结论：`const` 表示只读约束，不一定是编译期常量；`constexpr` 则强调可以在编译期求值。简单说：所有 `constexpr` 都是 const 语义，但反过来不成立。


详细解释：


- `const int x = input();` 是只读，但不是编译期常量。
- `constexpr int x = 42;` 可用于数组长度、模板参数等编译期场景。
- `constexpr` 还能修饰函数，表示其满足编译期求值条件。


常见坑/追问：


- `constexpr` 函数不代表一定在编译期执行，要看调用上下文。

### Q14: 🟡 什么是 POD、trivial、standard-layout 类型？


A: 结论：它们是描述类型内存与构造语义的标准术语，用来判断某些类型是否接近“纯数据”。现代标准中 POD 概念已被拆分，更常看 trivial 和 standard-layout。


详细解释：


- trivial：构造、拷贝、析构等行为接近编译器默认生成。
- standard-layout：内存布局满足一些规则，便于与 C 兼容。
- 这些性质影响序列化、共享内存、二进制拷贝等场景。


常见坑/追问：


- “能 `memcpy`” 不等于一切都安全，要看对象语义。

> 💡 **面试追问**：内存泄漏如何定位？Valgrind 和 AddressSanitizer 各自适合什么场景？



### Q15: 🟡 位运算在工程里有哪些典型应用？


A: 结论：位运算常用于权限标志、状态位、协议字段、性能敏感判断和低层数据编码。它不只是面试算法题，在通信和系统编程里很常见。


详细解释：


- 权限组合：读/写/执行位。
- 协议报文：从某个字节提取标志位。
- 设备状态：bitmask 表示多个布尔状态。


代码示例：


```cpp
enum Permission { Read = 1, Write = 2, Exec = 4 };
int p = Read | Write;
bool canWrite = p & Write;
```


常见坑/追问：


- 注意运算优先级与符号位扩展。
