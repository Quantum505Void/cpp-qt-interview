### Q1: ⭐🟢 什么是多态？


A: 结论：多态就是“同一接口，面对不同对象产生不同表现”。在 C++ 里，多态既包括编译期多态，也包括运行期多态。


详细解释：


- 编译期多态：重载、模板。
- 运行期多态：虚函数。
- 面试里通常重点问运行期多态，因为它涉及对象模型和 vtable。


常见坑/追问：


- 不要把“继承”直接等同于“多态”，没有虚函数不算运行期多态。


### Q2: ⭐🟡 编译期多态和运行期多态有什么区别？


A: 结论：编译期多态在编译时就确定调用目标，效率高；运行期多态在运行时根据对象真实类型决定调用目标，更灵活。它们本质是“静态绑定”和“动态绑定”的区别。


详细解释：


- 模板、函数重载属于静态分发。
- 基类指针/引用调用虚函数属于动态分发。
- 前者偏性能与泛型，后者偏扩展性与抽象接口。


常见坑/追问：


- 不同场景各有优势，面试时别说成“运行期多态更高级”。


### Q3: ⭐🟡 虚函数为什么能实现运行期多态？


A: 结论：因为编译器通常会为含虚函数的类维护虚函数表（vtable），对象内部有隐藏指针（vptr）指向该表，调用虚函数时通过 vtable 间接分发到真实类型实现。也就是说，多态不是魔法，是对象布局 + 间接调用机制。


详细解释：


- 基类接口只描述“可调用什么”。
- 具体调用哪个实现，由对象运行时的真实类型决定。
- 这也是虚函数会带来一点额外开销的原因。


常见坑/追问：


- vtable/vptr 属于常见实现，不是标准强制写法，但主流编译器基本如此。


### Q4: 🟡 vtable 和 vptr 分别是什么？


A: 结论：vtable 是类级别的虚函数表，vptr 是对象中的隐藏指针，用来指向对应类的虚函数表。调用虚函数时，实际过程通常是 `obj -&gt; vptr -&gt; vtable -&gt; function`。


详细解释：


- 每个多态类通常有一张虚表。
- 每个对象通常有一个 vptr。
- 多重继承、虚继承时对象布局会更复杂。


常见坑/追问：


- 面试时别把 vtable 说成“每个对象一张表”，那就翻车了。


### Q5: ⭐🟢 为什么基类析构函数通常必须是 virtual？


A: 结论：因为如果你通过基类指针删除派生类对象，而基类析构不是虚函数，就只会执行基类析构，导致派生类资源泄漏或行为未定义。对“设计成可被继承并通过基类接口使用”的类，虚析构几乎是标配。


详细解释：


- `delete basePtr;` 时需要动态绑定析构链。
- 没有虚析构，删除行为和对象真实类型不匹配。
- 这类问题在线上很隐蔽。


代码示例：


```cpp
class Base {
public:
    virtual ~Base() = default;
};
```


常见坑/追问：


- 不是所有类都必须虚析构，只有“打算多态使用”的基类才特别需要。


### Q6: ⭐🟡 纯虚函数和抽象类是什么关系？


A: 结论：类中只要有一个纯虚函数，通常它就是抽象类，不能被实例化。纯虚函数的作用是强制派生类提供某个接口实现。


详细解释：


- 纯虚函数写法：`virtual void f() = 0;`
- 抽象类适合表达接口或半抽象模板。
- 派生类如果没实现所有纯虚函数，也仍然是抽象类。


代码示例：


```cpp
class Shape {
public:
    virtual double area() const = 0;
    virtual ~Shape() = default;
};
```


常见坑/追问：


- 纯虚析构函数也可以存在，但仍然要提供定义。


### Q7: 🟡 构造函数为什么不能是虚函数？


A: 结论：因为对象在构造阶段还没完成，动态分发依赖的“完整对象类型”与虚机制基础都还未稳定，虚构造没有语义基础。说白了，对象都没活完整，谈不上通过它做多态分发。


详细解释：


- 构造的目标本身就是建立对象布局和 vptr。
- 如果构造还要靠虚机制决定自己是谁，会产生循环悖论。


常见坑/追问：


- 面试官常顺带问：析构函数为什么却可以是虚的？因为析构发生在对象已存在之后。


### Q8: ⭐🟡 构造/析构阶段调用虚函数会发生什么？


A: 结论：在构造和析构期间，虚函数不会表现为完整的动态多态，而只会调用当前阶段所属类版本。这是 C++ 对对象构造安全性的保护，也是经典面试坑。


详细解释：


- 基类构造期间，派生部分还没初始化完成。
- 基类析构期间，派生部分也已被销毁。
- 因此虚调用会被限制在当前构造/析构层级。


常见坑/追问：


- 不要在构造函数里依赖虚函数做关键初始化逻辑。


### Q9: 🟡 多态会带来哪些成本？


A: 结论：多态的成本主要包括对象额外 vptr、一次间接调用、部分场景下不利于内联和优化，以及更复杂的类层次维护。成本通常可接受，但在超热路径上要有意识。


详细解释：


- 对象大小可能增加一个指针。
- 调用要走间接跳转。
- 分支预测和 cache 局部性有时会受影响。


常见坑/追问：


- 不要把虚函数性能问题夸大成“几乎不能用”，绝大多数业务代码不是瓶颈。


### Q10: ⭐🟡 什么时候不建议用继承多态？


A: 结论：当对象关系不是 is-a、类层级不断膨胀、变化点主要在行为而不是身份时，往往不该优先继承多态，而应考虑组合、策略模式、模板或 `std::variant`。能不用深继承，就别把系统养成一棵圣诞树。


详细解释：


- 继承会引入强耦合。
- 子类数量一多，维护成本迅速上升。
- 组合通常更灵活、更易测试。


常见坑/追问：


- 回答时最好给一个场景例子，比如“多种算法切换更适合策略模式”。


### Q11: 🟢 基类指针和基类引用都能实现多态吗？


A: 结论：能，只要调用的是虚函数，基类引用和基类指针都可以触发运行期多态。二者主要区别在于是否允许为空和使用方式。


详细解释：


- 基类引用更适合表达“对象一定存在”。
- 基类指针可表达可空语义，也更方便做容器存储。


常见坑/追问：


- 不是“通过对象本身调用”就不能多态，而是对象切片会破坏多态语义。


### Q12: ⭐🟡 什么是对象切片（object slicing）？


A: 结论：对象切片是指把派生类对象按值赋给基类对象时，派生部分被截掉，只保留基类子对象。它会直接破坏多态语义，是值语义继承体系中的高频坑。


详细解释：


- 按值传参、按值返回、容器装基类对象都可能发生切片。
- 多态类型通常应通过指针、引用或智能指针使用。


代码示例：


```cpp
Derived d;
Base b = d; // slicing
```


常见坑/追问：


- 如果面试官追问，顺带讲为什么 `std::vector&lt;Base&gt;` 不能安全装 `Derived` 多态对象。
